<task type="auto">
  <name>Task 2: Create udev rule and install script</name>
  <files>
    install/99-kbd-backlight.rules

  <action>
    Create `install/99-kbd-backlight.rules` with the exact RUN+chgrp/chmod pattern. Do NOT use GROUP= or MODE= — these do not reliably apply to sysfs attribute files (confirmed pitfall from research). The rule must use `RUN+` commands explicitly.

    Content of `install/99-kbd-backlight.rules`:
    ```
    # Grant plugdev group write access to ASUS TUF keyboard RGB mode control.
    # Uses RUN+chgrp/chmod (not GROUP=/MODE=) — required for sysfs attribute files.
    # User must be a member of the plugdev group (Ubuntu default for hardware access).
    ACTION=="add", SUBSYSTEM=="leds", KERNEL=="asus::kbd_backlight*", \
      RUN+="/bin/chgrp plugdev /sys/class/leds/%k/kbd_rgb_mode", \
      RUN+="/bin/chmod g+w /sys/class/leds/%k/kbd_rgb_mode"
    ```

    Create `install/setup-permissions.sh` as a bash script that:
    1. Checks it is running as root (exit with error message if not)
    2. Copies `99-kbd-backlight.rules` to `/etc/udev/rules.d/99-kbd-backlight.rules`
    3. Runs `udevadm control --reload-rules`
    4. Runs `udevadm trigger --action=add /sys/class/leds/asus::kbd_backlight` to apply immediately without requiring a reboot
    5. Adds the current user (`$SUDO_USER`) to the `plugdev` group with `usermod -aG plugdev $SUDO_USER` (idempotent; documents the group requirement)
    6. Prints a success message instructing the user to log out and back in for the group membership to take effect (required for `$SUDO_USER` even though `hikami` is already in plugdev — makes the script correct for any user)

    The script must be placed in `install/setup-permissions.sh` and be executable (`chmod +x install/setup-permissions.sh`).

    The script should live in `install/` alongside the rule file. All `sudo` calls in the script assume it is already running as root (invoked via `sudo install/setup-permissions.sh`), so use `$SUDO_USER` for the original user identity.
  </action>
