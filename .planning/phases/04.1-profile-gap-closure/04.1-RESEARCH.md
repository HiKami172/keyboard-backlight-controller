## Summary

Phase 4.1 is a surgical gap-closure phase with exactly two changes identified by the v1.0 milestone audit. Both changes are tiny in scope and all required code, APIs, and patterns already exist in the codebase.

**GAP-1 (PROF-04):** `_restore_last_profile()` in `kbd_backlight/ui/application.py` is inside the `if not self._tray_only:` guard at line 74. When the app starts via XDG autostart (`--tray-only`), the keyboard stays at its firmware default state instead of restoring the user's last profile. The fix is a one-line move: extract `self._restore_last_profile()` to run before the `if not self._tray_only:` conditional so it executes on every first activation.

**GAP-2 (PROF-02):** `ProfileManager.rename_profile()` is fully implemented and has 100% test coverage (Phase 2). MainWindow (`kbd_backlight/ui/window.py`) exposes Save and Delete but has no Rename UI. The fix adds a Rename button row to `_build_profile_section()`, wires it to a dialog using the existing `Adw.Dialog` + `Adw.EntryRow` pattern (matching `_show_save_dialog()`), calls `manager.rename_profile()`, then calls `_refresh_profile_list()` and `notify_tray_refresh()` — exactly the same post-mutation housekeeping used by Save and Delete.

**Primary recommendation:** Implement as two plans: Plan 04.1-01 fixes PROF-04 (application.py, 1 line change + test), Plan 04.1-02 adds the rename UI (window.py, ~40 lines + test).

### Pattern 1: GAP-1 Fix — Move `_restore_last_profile()` Out of Conditional

**What:** In `application.py`, `_on_activate()` currently calls `_restore_last_profile()` only in the `if not self._tray_only:` branch. Moving it above the conditional ensures it runs on every first activation.

**Current code (broken):**
```python
# kbd_backlight/ui/application.py  lines 73–76
if not self._tray_only:
    self._window.present()
    self._restore_last_profile()
```

**Fixed code:**
```python
self._restore_last_profile()   # always restore on first activation (PROF-04)
if not self._tray_only:
    self._window.present()
```

**Important:** `_restore_last_profile()` uses `persist=False` (cmd=0), which is correct — it only restores the hardware display state without rewriting firmware. This is already the behavior in the existing implementation.

**When to use:** This is a one-off correction. No new pattern — just a placement correction.

### Pattern 2: GAP-2 Fix — Rename UI in MainWindow

**What:** Add a Rename button row to `_build_profile_section()`. When clicked, show an `Adw.Dialog` (matching `_show_save_dialog()` structure) pre-populated with the current profile name. On confirm, call `manager.rename_profile(old_name, new_name)`, then call `_refresh_profile_list()` and `notify_tray_refresh()`.

**Where:** `kbd_backlight/ui/window.py`, inside `_build_profile_section()`, between the Save row and Delete row.

**New Rename row in `_build_profile_section()`:**

```python
# After save_row, before delete_row — within _build_profile_section()
rename_row = Adw.ActionRow()
rename_row.set_title('Rename Profile')
rename_row.set_subtitle('Change the name of the selected profile')
rename_btn = Gtk.Button(label='Rename…')
rename_btn.set_valign(Gtk.Align.CENTER)

rename_row.add_suffix(rename_btn)
```

**`_on_rename_clicked` handler:**
```python
def _on_rename_clicked(self, _btn):
    idx = self._profile_row.get_selected()
    if idx == Gtk.INVALID_LIST_POSITION:
        return
    names = self._manager.list_profiles()
    if not names or idx >= len(names):
        return
    old_name = names[idx]
    self._show_rename_dialog(old_name)
```

**`_show_rename_dialog` — mirrors `_show_save_dialog` pattern:**
```python
def _show_rename_dialog(self, old_name: str):
    dialog = Adw.Dialog.new()
    dialog.set_title('Rename Profile')
    dialog.set_content_width(360)

    box = Gtk.Box(orientation=Gtk.Orientation.VERTICAL, spacing=12)
    box.set_margin_top(12)
    box.set_margin_bottom(12)
    box.set_margin_start(12)
    box.set_margin_end(12)

    entry = Adw.EntryRow.new()
    entry.set_title('New Name')
    entry.set_text(old_name)  # pre-populate with current name
    box.append(entry)

    button_box = Gtk.Box(orientation=Gtk.Orientation.HORIZONTAL, spacing=8)
    button_box.set_halign(Gtk.Align.END)
    cancel_btn = Gtk.Button(label='Cancel')
    rename_btn = Gtk.Button(label='Rename')
    rename_btn.add_css_class('suggested-action')
    cancel_btn.connect('clicked', lambda _: dialog.close())
    rename_btn.connect('clicked', lambda _: self._do_rename(old_name, entry.get_text(), dialog))
    button_box.append(cancel_btn)
    button_box.append(rename_btn)
    box.append(button_box)

    dialog.set_child(box)
    dialog.present(self)
```

**`_do_rename` — mutation + housekeeping:**
```python
def _do_rename(self, old_name: str, new_name: str, dialog):
    new_name = new_name.strip()
    if not new_name or new_name == old_name:
        dialog.close()
        return
    try:
        self._manager.rename_profile(old_name, new_name)
    except ValueError:
        # new_name already exists — show error toast, keep dialog open
        self._toast_overlay.add_toast(
            Adw.Toast.new(f'A profile named "{new_name}" already exists')
        )
        return
    self._refresh_profile_list()
    self.get_application().notify_tray_refresh()
    self._toast_overlay.add_toast(Adw.Toast.new(f'Renamed to "{new_name}"'))
    dialog.close()
```

**Note on `manager.rename_profile()` error contract:**
- `KeyError` if `old_name` not found — should not happen if UI only calls this for selected profile, but guard is prudent.
- `ValueError` if `new_name` already exists — must show user feedback and not close the dialog.

### Pattern 3: Tray Refresh After Rename

After rename, the tray menu must be rebuilt so the profile shows under its new name. This uses the existing `notify_tray_refresh()` path already used by Save and Delete. No new IPC is needed — `REFRESH` is the existing command.

```python
# Already in application.py:
def notify_tray_refresh(self):
    self._send_tray('REFRESH')
```

### Anti-Patterns to Avoid

- **Closing the dialog on `ValueError`:** If the user enters a name that collides with an existing profile, the dialog must stay open and show a toast. Closing it forces the user to reopen and re-type the new name.
- **Not pre-populating the rename entry:** Starting the rename dialog with an empty field forces the user to re-type the full profile name. Pre-populate with `entry.set_text(old_name)`.
- **Calling `_restore_last_profile()` only in the non-tray branch:** This is the existing bug. The fix is to move the call before the conditional entirely — do not add a second call inside the tray branch.
- **Using `persist=True` in `_restore_last_profile()`:** The existing implementation uses `persist=False` (cmd=0) — this is correct. Restore must not rewrite firmware; it should only set the display state.
- **Forgetting `_refresh_profile_list()` and `notify_tray_refresh()` after rename:** Both the dropdown and tray menu show stale names if not refreshed. Use the same housekeeping chain as Save and Delete.

## Common Pitfalls

### Pitfall 1: `_restore_last_profile()` Blocking Startup
**What goes wrong:** `_restore_last_profile()` calls `BacklightController.apply()`, which does a sysfs write. If hardware is unavailable, this raises an exception. The existing implementation wraps the `apply()` call in `try/except` and continues — this is the correct behavior. Do not change the error handling.
**How to avoid:** Keep the `except Exception: pass` in `_restore_last_profile()`. The docstring says "Hardware unavailable; don't block startup."
**Warning signs:** Any change that removes the try/except would cause startup failures when hardware is not found.

### Pitfall 2: Rename Dialog Closing on `ValueError` (Collision)
**What goes wrong:** `_do_rename()` calls `dialog.close()` even when `manager.rename_profile()` raises `ValueError`. The dialog disappears and the user's input is lost.
**Why it happens:** Calling `dialog.close()` unconditionally after the try/except block.
**How to avoid:** `return` early after showing the collision toast; only call `dialog.close()` on success.
**Warning signs:** Dialog disappears when user tries a colliding name; user must reopen and retype.

### Pitfall 3: `_on_rename_clicked` Called When No Profiles Exist
**What goes wrong:** If the profile list shows `(no profiles)` (the placeholder), `idx` maps to index 0 but `names` is empty. Accessing `names[0]` raises `IndexError`.
**Why it happens:** `_refresh_profile_list()` inserts `['(no profiles)']` when empty, so `get_selected()` returns 0 even though no real profile exists.
**How to avoid:** Guard with `if not names or idx >= len(names): return` — same guard used by `_on_delete_clicked` and `_on_profile_selected`.
**Warning signs:** `IndexError` in `_on_rename_clicked` when no profiles are saved.

### Pitfall 4: Tray Menu Shows Old Name After Rename
**What goes wrong:** `notify_tray_refresh()` is not called after rename, so the tray menu still shows the old profile name.
**Why it happens:** Forgetting to follow the same post-mutation housekeeping as Save and Delete.
**How to avoid:** Call `self.get_application().notify_tray_refresh()` immediately after `self._refresh_profile_list()` in `_do_rename()`.
**Warning signs:** Tray menu shows old name; clicking old entry fails silently (profile not found by old name).

### Pitfall 5: Rename Entry Not Pre-Populated
**What goes wrong:** The rename dialog opens with an empty entry, requiring the user to type the full profile name from scratch.
**How to avoid:** Call `entry.set_text(old_name)` when building the rename dialog.
**Warning signs:** Rename dialog opens blank.

### Pitfall 6: `_activated_once` Guard and Restore Ordering
**What goes wrong:** `_restore_last_profile()` depends on `self._controller` being initialized. If moved before controller init, it will call apply on `None`.
**Why it happens:** Careless reordering of the `_on_activate()` body.
**How to avoid:** The correct placement for `_restore_last_profile()` is after the controller and window are initialized, but before (or after) the `if not self._tray_only:` window presentation block. In the current code, all of this is after controller init. The one-line fix moves it from inside the `if not self._tray_only:` block to just before it — not before controller init.
**Warning signs:** `AttributeError: 'NoneType' object has no attribute 'apply'` on startup.

```python
# Source: kbd_backlight/ui/application.py — existing implementation
def _restore_last_profile(self):
    """Apply last-used profile to hardware on startup (PROF-04)."""
    profile = self._manager.get_last_profile()
    if profile is None:
        return
    try:
        self._controller.apply(
            profile.mode, profile.r, profile.g, profile.b,
            profile.speed, persist=False,  # cmd=0 — restore display only
        )
    except Exception:
        pass  # Hardware unavailable; don't block startup
```

### `manager.rename_profile()` API (already implemented in Phase 2)

```python
# Source: kbd_backlight/profiles/manager.py
def rename_profile(self, old_name: str, new_name: str) -> None:
    """Rename a profile. Raises KeyError (not found) or ValueError (collision)."""
    data = self._load()
    if old_name not in data["profiles"]:
        raise KeyError(f"Profile '{old_name}' not found")
    if new_name in data["profiles"]:
        raise ValueError(f"Profile '{new_name}' already exists")
    profile_data = data["profiles"].pop(old_name)
    profile_data["name"] = new_name
    data["profiles"][new_name] = profile_data
    if data.get("last_profile") == old_name:
        data["last_profile"] = new_name
    self._save(data)
```

## Open Questions

1. **Should rename use a confirmation dialog or act directly?**
   - What we know: Save acts directly (no confirmation). Delete uses `Adw.AlertDialog` because it is irreversible.
   - What's unclear: Whether rename warrants a confirmation step.
   - Recommendation: No confirmation. Rename is low-stakes and easily undone by renaming again. Matching the Save dialog's direct action is correct. Show success toast instead of a confirmation.

2. **Where exactly in `_build_profile_section()` should the Rename row appear?**
   - What we know: Current order is: Profile selector (ComboRow) → Save row → Delete row.
   - Recommendation: Insert Rename between Save and Delete — the semantic order becomes "Create/Load → Save → Rename → Delete", progressing from less destructive to more destructive.

3. **Should Rename be disabled when no profiles exist (list shows placeholder)?**
   - What we know: Save button is always enabled (creates a new profile). Delete button is always enabled but guarded at click time.
   - Recommendation: Apply the same guard-at-click approach used by Delete — let the button always be enabled, but return early in `_on_rename_clicked` if `names` is empty. This is consistent with existing UI behavior and avoids complexity of disabling/re-enabling the button reactively.

4. **Test coverage for PROF-04 fix — is a unit test needed for `_restore_last_profile()` placement?**
   - What we know: `application.py` has no existing unit tests (GTK app testing requires a running event loop or mock). `_restore_last_profile()` itself is straightforward.
   - Recommendation: The most practical verification is a targeted integration check: after the fix, run the app with `--tray-only` and confirm the keyboard hardware reflects the last profile. Automated unit test for application startup flow may not be worth the GTK mocking complexity for a one-line fix. Document the test approach in the plan.
