### 1. Autostart hardware restore (PROF-04 integration)

**Expected:** Keyboard immediately shows the last-used profile's color and mode rather than the firmware default (typically static white or off)
**Why human:** Requires real hardware and real sysfs writes; the `_restore_last_profile()` call path cannot be exercised without a running GLib main loop and a real `BacklightController`

### 2. Rename collision — dialog stays open

**Test:** Open app, save two profiles named "Work" and "Night", select "Work" in the dropdown, click Rename, type "Night", click Rename button
**Expected:** A toast appears saying "A profile named 'Night' already exists" and the rename dialog remains open (not dismissed)
**Why human:** GTK4 widget state (dialog visibility, toast display) requires a running event loop; cannot verify that `return` in the `except ValueError` block actually prevents `dialog.close()` from being called by verifying widget state

### 3. Post-rename dropdown and tray refresh (PROF-02 integration)

**Test:** Rename a profile, then observe the profile ComboRow dropdown and right-click the system tray icon
**Expected:** The ComboRow shows the new name immediately; the tray menu also shows the new name
**Why human:** The tray is a subprocess (`tray.py`); whether `notify_tray_refresh()` successfully causes the menu to rebuild requires observing the tray subprocess response
